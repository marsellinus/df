Abstract

This research presents a forensic reconstruction of an unauthorized data exfiltration attack targeting a private cloud storage system (Nextcloud 29) deployed in a Dockerized environment. The study addresses the research gap in distributed metadata analysis for detecting and reconstructing exfiltration attacks across multiple log sources. The primary objective is to develop and validate a Distributed Metadata Correlation Engine capable of identifying the actor, timeline, method, and volume of exfiltrated data from real system-generated logs.

The methodology follows the NIST Digital Forensic Process framework, incorporating log ingestion from nginx access logs and Nextcloud audit logs, metadata correlation across sources, rule-based anomaly detection (five detectors), risk scoring (0–100), and chronological timeline reconstruction with attack-phase labeling. Experimental results demonstrate that the system successfully identified the primary suspect (alice) with a risk score of 76/100 (HIGH), detected all 7 sensitive files exfiltrated (1.26 MB total), and reconstructed the complete attack timeline within a 24-second exfiltration window. The contribution of this research is a modular, reproducible forensic pipeline that produces structured evidence output (who/when/what/how/how much) from real cloud storage logs without reliance on synthetic datasets.

Keywords: digital forensics, data exfiltration, cloud storage, metadata analysis, Nextcloud, log analysis, anomaly detection, timeline reconstruction

1. Introduction

The rapid adoption of private cloud storage solutions has introduced significant security challenges, particularly regarding unauthorized data exfiltration. Organizations increasingly rely on platforms such as Nextcloud to store sensitive documents, yet the forensic investigation of exfiltration incidents in these environments remains underexplored. Traditional forensic tools are designed for endpoint or network-level analysis and do not adequately address the distributed, multi-source log structure of cloud storage systems.

The research problem addressed in this study is the difficulty of reconstructing a complete attack narrative from fragmented log sources in a private cloud environment. Existing approaches either rely on network packet capture (which may be unavailable in containerized deployments) or single-source log analysis (which misses cross-source correlations). This creates a research gap: no unified pipeline exists that correlates metadata across nginx access logs, application audit logs, and file-access events to produce a structured forensic reconstruction.

The objectives of this research are: (1) to design and implement a Distributed Metadata Correlation Engine for private cloud storage forensics; (2) to validate the system against real attack traffic generated on a live Nextcloud instance; and (3) to produce a structured forensic output answering the five investigative questions: who, when, what, how, and how much.

The contribution of this work is a modular, open-source forensic pipeline that operates entirely on system-generated logs, requires no network capture, and produces reproducible results. The system is validated against a real exfiltration scenario executed on a Dockerized Nextcloud 29 environment.

2. Related Work / Literature Review

Digital forensics in cloud environments has been studied from multiple perspectives. Zawoad and Hasan [1] identified the challenges of cloud forensics, including multi-tenancy, data volatility, and jurisdictional issues, establishing the foundational problem space for this research. Their work highlights that traditional forensic models are insufficient for cloud-native environments.

Regarding log-based forensics, Marty [2] demonstrated that structured logging (JSON format) significantly improves forensic analysis quality compared to unstructured text logs. This finding directly informs our use of Nextcloud's JSON audit log format and nginx's forensic_json log format.

In the domain of anomaly detection for exfiltration, Homoliak et al. [3] surveyed insider threat detection methods and identified bulk download patterns, off-hours access, and rapid succession requests as the most reliable behavioral indicators. Our five-detector anomaly engine implements these exact indicators.

Timeline reconstruction in digital forensics has been formalized by Olsson and Boldt [4], who proposed a model for correlating events across multiple sources into a unified chronological narrative. Our timeline reconstructor extends this model with attack-phase labeling (EXFILTRATION, TARGET_ACCESS, RECONNAISSANCE, etc.).

The NIST Digital Forensic Process [5] provides the methodological framework adopted in this research: identification, collection, examination, analysis, and reporting. Our pipeline maps directly to these phases.

Prior work on Nextcloud-specific forensics is limited. Studiawan et al. [6] analyzed log formats in cloud storage applications but did not address cross-source metadata correlation. This gap motivates the Distributed Metadata Correlation Engine proposed in this study.

2.1 Relevant Concepts

WebDAV (Web Distributed Authoring and Versioning) is the protocol used by Nextcloud for file access via HTTP. All file download operations appear as HTTP GET requests to /remote.php/dav/files/<user>/<filename>, making nginx access logs the primary evidence source for exfiltration detection.

Metadata correlation refers to the process of combining attributes from multiple log sources (timestamps, user identifiers, IP addresses, file names, byte counts) to build a unified behavioral profile per actor. This is distinct from single-source analysis and enables detection of patterns that span log boundaries.

3.3 Data Acquisition

Two log sources are collected from the running Docker containers:

  • nginx access log (logs/nginx/access.log) — JSON format with fields: timestamp, remote_addr, request_method, request_uri, status, body_bytes_sent, user_agent. This is the primary source for file download evidence as it records actual byte counts transferred.
  • Nextcloud admin_audit log (logs/nextcloud/nextcloud.log) — JSON format with fields: time, user, method, url, message, remoteAddr, userAgent. This source provides authentication events (login attempts, successes) and file operation events.

Logs are shared between containers via Docker bind-mount volumes, ensuring the forensic dashboard can read them without network transfer. Log integrity is preserved by treating the files as read-only inputs to the analysis pipeline.

4. Experiment / Investigation Process

4.1 Attack Scenario Design

The attack scenario simulates a compromised user account performing unauthorized bulk exfiltration of sensitive corporate files. The scenario is structured in five phases to reflect real-world attacker behavior:

  • Phase 1 – Reconnaissance: Three failed login attempts using incorrect passwords to test credential validity.
  • Phase 2 – Enumeration: PROPFIND request to list all files in the victim's directory.
  • Phase 3 – Bulk Download: Sequential GET requests for all 7 sensitive files in rapid succession (0.3s intervals).
  • Phase 4 – Verification: Re-download of 3 files (financial_report, api_keys, customer_database) to confirm successful exfiltration.
  • Phase 5 – Tool Switching: Same files accessed using different User-Agent strings (Wget, python-requests) to evade simple UA-based detection.

4.3 Pipeline Execution

After the attack, logs were copied from the Docker containers to the host filesystem and ingested into the forensic pipeline using the following command sequence:

# Copy real logs from containers cp logs/nginx/access.log logs/nc_nginx_access.log # Ingest into forensic database python main.py --ingest logs/nc_nginx_access.log # Run full pipeline and generate report python main.py --output data/forensic_report.json

The pipeline executed the following steps automatically: (1) parse and normalize 213 log records from nginx JSON format; (2) build per-user and per-IP metadata profiles; (3) run five anomaly detectors; (4) compute Exfiltration Risk Score per user; (5) reconstruct chronological timeline with phase labels; (6) generate structured forensic report.

6. Discussion

6.1 Strengths of the Method

The Distributed Metadata Correlation Engine demonstrated several strengths. First, by combining nginx access logs (which contain actual byte counts) with Nextcloud audit logs (which contain authentication and file-operation events), the system achieves higher detection accuracy than single-source analysis. The nginx log alone is sufficient to detect the exfiltration; the audit log adds authentication context (failed logins, login success) that enables reconstruction of the reconnaissance phase.

Second, the risk scoring model correctly ranked alice as the primary suspect (score 76) while keeping normal administrative activity (admin, score 33) below the alert threshold. The severity bonus mechanism — which adds 25 points for CRITICAL anomalies — ensures that users with confirmed sensitive file access are elevated appropriately.

Third, the system operates entirely on system-generated logs without requiring network packet capture, making it deployable in containerized environments where packet capture is impractical.

6.2 Limitations

The current implementation has several limitations. The off-hours detector flagged all users (alice, admin, unknown) because the experiment was conducted at 03:41 UTC, which is outside business hours for all users. In a production deployment, per-user baseline hours would need to be established to reduce false positives. Additionally, the system does not currently correlate with external threat intelligence (e.g., known malicious IPs) or perform geolocation analysis, which would strengthen the attribution confidence.

The bulk download threshold (5 GET operations) may generate false positives for legitimate power users who regularly download large numbers of files. A sliding-window approach with per-user baseline comparison would improve precision.

6.3 Research Implications

This research demonstrates that distributed metadata correlation is a viable and effective approach for cloud storage forensics. The modular pipeline architecture allows individual components (parsers, detectors, scoring weights) to be adapted for other cloud platforms (MinIO, ownCloud, S3) with minimal modification. The structured output format (who/when/what/how/how much) aligns with standard forensic reporting requirements and can be directly used as evidence in incident response procedures.

7. Conclusion

This research successfully designed, implemented, and validated a Distributed Metadata Correlation Engine for forensic reconstruction of unauthorized data exfiltration in private cloud storage. The system was tested against a real attack scenario on a live Nextcloud 29 instance and produced accurate, structured forensic output.

The main findings are: (1) the primary suspect (alice) was correctly identified with a risk score of 76/100; (2) all 7 sensitive files totaling 1.26 MB were confirmed exfiltrated; (3) the complete attack timeline was reconstructed with accurate phase labeling; (4) the system detected automated tool usage through rapid-succession pattern analysis.

The contribution of this research is a reproducible, modular forensic pipeline that operates on real system-generated logs, requires no synthetic data, and produces evidence-grade output. Future work should address per-user behavioral baselines, integration with external threat intelligence, and extension to other cloud storage platforms.
